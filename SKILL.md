# aider-stack

Orient Claude for handling instructions generated by ai-watch.

## When to Use
- Triggered by running `/aider-stack` or when asked to "use aider mode".

## Instructions
1. When this skill is triggered **check for plan**: Check for `PLAN.md` or `docs/PLAN.md` in the project's root directory, which may differ from `$PWD` (e.g. `root=$(git rev-parse --show-toplevel); test -f "$root/PLAN.md" || test -f "$root/docs/PLAN.md"`). If either file exists, read its contents for context and keep it in mind, but **do not carry out the plan now** — only prepare to act on the dispatch markers in subsequent user prompt.
2. When this skill is triggered suggest the first step to do.
3. On subsequent user prompts, whenever dispatch markers are encountered:

	* **Restrict Edits**: Make edits **ONLY** in the files explicitly mentioned for edit.
	* **Propose Other Files**: If you believe it is useful or want to edit other files, do not modify them. Instead, *suggest* using those files.
	* **Brief Output**: Keep all responses and explanations brief.
	* **Suggest Next Step** (DO only): When finished, if there is a `DO` heading in the prompt — suggest the next logical step, see below.

## Prompts containing dispatch markers
A dispatch marker prompt may cover several files. Each file is marked either `EDIT:` (you may edit it) or `FILE:` (reference only — do not edit it), followed by its items under the headings below.
- **DO**: Make the change described at the referenced lines.
- **INLINE**: Insert a new comment line directly below the referenced line, using the file's comment syntax, of the exact form `AI?> <your reply>`.
- **ASK**: Reply to the question in the chat window. Do not modify the file.
- **CONTEXT**: Treat the block as information for your task. Do not modify the file.

## Suggested First/Next Step
This should be a bulleted (•) list of 1-3 *related* items. Each one (if possible) should start with a **full file path relative to the $PWD** and a brief suggestion for user to follow. The suggestion should be contextual and be related to the PLAN.md and the current state of the implementation. For instance, if the plan mentions correcting bugs and adding configuration:
    • `./BUGS.md` - contains bugs which we should revise
    • `./src/main/config.cpp` - create this configuration file
